Referencia completa de Conventional Commits para Git

tipos de commit más usados

docs:

Cambios en documentación. README, comentarios en el código, guías de uso.

docs: add README with project overview
feat:

Nueva funcionalidad (feature). Algo que antes no existía en el proyecto.

feat: add real-time gesture detection with GAT
fix:

Corrección de un error (bug). El código existía pero fallaba.

fix: resolve batch_size mismatch in DataLoader
chore:

Mantenimiento que no afecta el código funcional. Configuraciones, dependencias, .gitignore.

chore: add Python .gitignore with project rules
refactor:

Mejorar el código sin cambiar lo que hace. Más legible, más eficiente, mismos resultados.

refactor: extract landmark normalization to utils
test:

Agregar o modificar pruebas. No cambia el código de producción.

test: add unit tests for graph builder function
perf:

Mejora de rendimiento. El código hace lo mismo pero más rápido o con menos memoria.

perf: move model inference to GPU with .cuda()
style:

Formato del código: espacios, indentación, punto y coma. Sin cambios de lógica.

style: reformat utils.py with consistent spacing

estructura de un commit

tipo: descripción corta en inglés
Usa inglés — tu repositorio es público y mundial
Máximo 72 caracteres en la descripción
Verbo en presente: "add", "fix", "remove" — no "added" ni "adding"
Evita mensajes vagos: "changes", "update", "fix stuff"
Un commit = una sola cosa. No mezcles feat + fix en el mismo commit
